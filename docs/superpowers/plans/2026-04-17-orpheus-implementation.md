## Guiding principles

1. **TDD**: every non-trivial module starts from a failing test with a known answer (BIP test vectors, known fixture wallets). Implement until green, then commit.
2. **No real secrets**: the `/Users/ddaniels/Dropbox/bitcoin/` scripts are *reference only* — copy logic, never key/password/wallet data. Fixtures are synthesized from freshly-generated throwaway entropy.
3. **Offline by default**: balance checker has a `mock` provider; demo fixtures only resolve against mock so `orpheus demo` runs air-gapped.
4. **Commit often**: each numbered step produces a commit.

## Phase 3 — Crypto primitives

Every step in this phase uses **known BIP test vectors** so correctness is checkable offline.

**3.1** `keys.py` — `privkey_to_pubkey`, `pubkey_to_p2pkh_address`, `privkey_to_wif`, `wif_to_privkey`, `validate_secp256k1_scalar`. Tests use BIP32 vector 1 seed → known addresses.
**3.2** `bip32.py` — HD derivation (`derive_path("m/44'/0'/0'/0/0")`). Tests use BIP32 test vector 1 & 2.
**3.3** `bech32.py` — segwit v0/v1 encode/decode. Tests use BIP173 + BIP350 vectors.
**3.4** Address helpers: `all_addresses_for_privkey(priv)` returns dict of p2pkh-compressed, p2pkh-uncompressed, p2sh-p2wpkh, bech32.
**3.5** Commit each submodule separately.

## Phase 4 — Extractors

Base interface: `class Extractor(ABC)` with `can_handle(path) -> bool` and `extract(path, passwords=None) -> list[ExtractedKey]`.

**4.1** `extractors/base.py` + registry (`list_extractors()`, `find_extractor(path)`).
**4.2** `wallet_dump.py` — parses Bitcoin Core `dumpwallet` text output. Tests use hand-crafted dump fixture.
**4.3** `bitcoin_core.py` — BDB + SQLite scanner using DER pattern `\x30\x81\xD3\x02\x01\x01\x04\x20` (from `extract_bdb_keys.py`). Tests use synthesized BDB fixture (generated by fixture builder, see Phase 8).
**4.4** `multibit.py` — protobuf parser finding IV(16)+encrypted(48) pairs; unencrypted path extracts raw privkeys. Tests use synthesized MultiBit fixture.
**4.5** `bip39.py` — mnemonic → seed → derive common paths (BIP44/49/84 + Breadwallet `m/0'/0/x`, `m/0'/1/x`). Tests use the BIP39 reference vector `"abandon abandon ... about"` with known addresses. **Breadwallet path is a first-class case — regression test anchors the 2026-04-17 recovery logic.**
**4.6** `blockchain_com.py` — V2 (1626-word) / V3 (65591-word) / V4 (GUID) / V5 (timestamp) decoders. Tests use hand-crafted vectors (word list checksums committed).
**4.7** `encrypted.py` — AES-256-CBC with scrypt or EVP_BytesToKey; iterates password list. Tests use known-plaintext encrypted fixture.
**4.8** Commit each extractor separately.

## Out of scope for v0.1

- Hardware wallet integration (Trezor/Ledger)
- Ethereum / other altcoin extraction (architecture allows; save for v0.2)
- GPU-accelerated password cracking (defer to btcrecover)
- Live HD wallet watching / automatic gap-limit expansion beyond the configured scan width

## Assumptions

1. 🟢 User runs on macOS/Linux with Python 3.10+ and `uv` installed (or via Docker).
2. 🟢 Demo fixtures can be synthesized deterministically from a seeded RNG so tests are reproducible.
3. 🟡 Synthesizing a "BDB-shaped" fixture is enough to exercise the DER-pattern extractor — if not, we'll bundle a real empty BDB wallet.dat generated by a one-off Bitcoin Core run (no keys of value).
4. 🟢 Blockchain.info / blockstream.info batch APIs remain stable (fallbacks between the two).
5. 🟡 BIP32 library quality: we'll hand-roll minimal BIP32 (no HD hardening surprises) rather than depend on `bip32` PyPI pkg which is less maintained.
